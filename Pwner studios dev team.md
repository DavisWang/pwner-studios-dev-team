# Initial prompt

I want to start leveraging harness engineering ([https://openai.com/index/harness-engineering/](https://openai.com/index/harness-engineering/)) more in my projects. As you can see from the games I create ([https://daviswang.github.io/pwner-studios/](https://daviswang.github.io/pwner-studios/)), game projects locally (Dots/Burn the village/Age of War/Diggr, etc), I spend a lot of time writing the initial spec, then play testing and iterating on logic/UI bugs, and expanding on gameplay mechanics. I want this to be more of a closed loop so subagents can keep each others accountable with much fewer directions from me. I want to create a team of subagents specifically for the type of indie game development I want to do. The output is a suggestion of how I can implement these subagents using harness engineering principles that work with each other seamlessly and autonomously that is LLM agnostic, I think the output is basically .md files like skills, but let me know if I'm wrong on this point. This is a new empty project deliberately, so the harness/agent teams that are created here can be reused by other LLMs and new game projects.

At a minimum. I want the following subagents:

- **Producer**  The producer is the orchestrator of every game project. Every game project starts with an initial spec from me. The producer is responsible for taking my initial spec and vision and augmenting the spec into clear and actionable artifacts for each subagent (noted below), then coordinating tasks across all subagents and ensuring the final output is of high quality and aligns with my vision and the initial spec. The producer is also responsible for deciding whether to surface subagent escalations to me, for example if any of the subagents feel like they require guidance/direction from me. Also, the producer is responsible for producing a TODOs/future directions artifact with feedback from all subagents below, the purpose of this artifact is to keep in a source of truth document how to improve the game in all aspects in the future, so that if another agent takes over development for this project, they can get up and running quickly.  
- **Game Designer**  The Game designer subagent is an expert in designing games. The game designer takes the initial spec as input and produces a polished final game spec ready for other subagent implementation. Getting to a final spec from the initial spec will take iteration with me, this is fine. The role of the game designer here is to 1 help clarify the spec where it’s ambiguous and get direction/guidance from me, and 2 help surface areas that should be considered to me to ensure the final spec can lead to a few shot polished output game.  
- **Game Developer**  The game developer is the agent responsible for producing the codebase for the game based on the final game spec. This includes: 1 architecture decisions (what stack to use), 2 implementing scaffolding/folder structures/etc to ensure extensible and reusable components, 3 writing the actual game code 4 writing detailed documentation both in the code and in documentation artifacts that describe the decisions/contexts that have been made in building the game, the goal here is to ensure that if another brand new agent takes over development of the project, that they can ramp up quickly without my guidance.  
- **Play Tester**  The play tester takes the final game spec and writes a testing plan with acceptance criteria/tests for the game, including 1 tests for logic, 2 UI/visual tests, and 3 integration tests. 4 game performance/latency tests. The play tester works with both the game developer and visual/interaction designer to ensure the requirements of the final spec is strictly adhered to.

Grown up in the early 90s/2000s, they miss the nostalgic feel of retro games they played when they were younger and are now re-experiencing them again as an adult.
